## JWT認証の仕組み

まず、JWTは「サーバー側でセッション情報を保持しなくても、トークンっ事態に情報を詰め込んで検証できる」認証方式で、これが**ステートレス**と呼ばれる理由です。

通常のセッション認証が、セッションIDをDBやメモリに保存して、リクエストごとに照合する（ステートフル）に対して、
JWT認証は、トークン事態に必要な情報を含め、署名によって改ざんされていないことを検証するだけでよい


### JWT構造

JWTは以下の3つの`.`でつないだ文字列
```
xxxxx.yyyyy.zzzzz
ヘッダー.ペイロード.署名
```

①Header
トークンのメタ情報。 
```
{
  "alg": "HS256",
  "typ": "JWT"
}
```
`alg`: 署名に扱うアルゴリズム
`typ` トークンの種類

②Payload
実際のデータ（クレーム）を格納する部分。
クレームとはユーザーやトークンに関する情報のこと。
```
{
  "sub": "1234567890",
  "name": "Taro Yamada",
  "iat": 1710000000,
  "exp": 1710003600
}
```

ペイロードはBase64URLでエンコードされているだけで、暗号化はされていないので、機密情報を入れてはいけない。

③Signature
ヘッダーとペイロードが改ざんされていないことを保証する部分。
```
HMACSHA256(
  base64UrlEncode(header) + "." + base64UrlEncode(payload),
  秘密鍵
)
```
サーバーはこの署名を検証することで「発行者が正しいか」「途中で中身が書き換えられていないか」を確認する。
